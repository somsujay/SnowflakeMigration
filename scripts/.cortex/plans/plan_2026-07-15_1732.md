## DevOps Strategy: QA → Pre-PROD → PROD via GitHub Actions

### Context

**Current state:**
- Existing workflow (`deploy.yml`) covers DEV (branch `ssom_gd`) and QA (branch `main`) only
- `environments.yml` defines only `dev` and `qa` databases
- Scripts use `snow sql` CLI with a single connection profile (`HAKKODAINC_PARTNER`)
- No approval gates, rollback strategy, or environment protection rules
- No Pre-PROD or PROD environments configured
- Smoke tests exist but are basic (schema/table/proc existence checks)

**Key findings:**
- `create_objects.sh` is hardcoded to a single DB — needs parameterization via `--env` flag
- The existing `deploy.sh` (referenced in CI but not present) needs to be created
- SQL scripts use `USE DATABASE` / `USE SCHEMA` commands that need to be parameterized per environment

---

### Branching & Promotion Strategy

```
feature/* ──► develop ──► release/* ──► main
                │              │            │
              [DEV]          [QA]      [Pre-PROD] ──► tag v* ──► [PROD]
              (auto)        (auto)    (auto+gate)           (manual approval)
```

| Environment | Trigger | Database | Approval |
|-------------|---------|----------|----------|
| DEV | Push to `develop` | `FINANCE_CORE_DEV` | None |
| QA | Push to `release/*` | `FINANCE_CORE_QA` | None (auto on merge) |
| Pre-PROD | Push to `main` | `FINANCE_CORE_PREPROD` | 1 reviewer required |
| PROD | Git tag `v*` on `main` | `FINANCE_CORE_PROD` | 2 reviewers required |

### Implementation Steps

#### 1. Update `environments.yml` — add Pre-PROD and PROD

```yaml
dev:
  database: FINANCE_CORE_DEV
  warehouse: SSOM_COCO_WH
  connection: HAKKODAINC_PARTNER

qa:
  database: FINANCE_CORE_QA
  warehouse: SSOM_COCO_WH
  connection: HAKKODAINC_PARTNER

preprod:
  database: FINANCE_CORE_PREPROD
  warehouse: SSOM_COCO_WH
  connection: HAKKODAINC_PARTNER

prod:
  database: FINANCE_CORE_PROD
  warehouse: PROD_WH
  connection: HAKKODAINC_PARTNER
```

#### 2. Create `scripts/deploy.sh` — environment-aware deployment script

Accepts `--env=dev|qa|preprod|prod`, reads `environments.yml`, substitutes database names in SQL scripts, and executes them in order (01–11). Includes:
- Dry-run mode (`--dry-run`) for PROD safety
- Idempotent `CREATE OR REPLACE` / `CREATE IF NOT EXISTS` validation
- Exit on first failure with clear error reporting

#### 3. Replace single workflow with a multi-stage pipeline

**New file:** `.github/workflows/ci.yml` (lint + unit validation on all PRs)

```yaml
# Triggers: pull_request to develop, release/*, main
# Jobs: sql-lint, schema-validation (compile-only checks)
```

**New file:** `.github/workflows/deploy-qa.yml`

```yaml
# Trigger: push to release/* branches
# Jobs: deploy → smoke-test → integration-test
# Environment: qa (no protection rules)
```

**New file:** `.github/workflows/deploy-preprod.yml`

```yaml
# Trigger: push to main
# Jobs: deploy → smoke-test → regression-test
# Environment: preprod (requires 1 approval via GitHub Environment protection)
```

**New file:** `.github/workflows/deploy-prod.yml`

```yaml
# Trigger: tag push matching v*
# Jobs: deploy → smoke-test → data-validation → notify
# Environment: production (requires 2 approvals, restricted to main branch)
# Includes: rollback job (manual trigger via workflow_dispatch)
```

#### 4. Create `scripts/rollback.sh`

- Accepts `--env` and `--version` (git tag or commit SHA)
- Checks out the specified version's SQL scripts
- Re-deploys using `CREATE OR REPLACE` semantics
- Runs smoke tests post-rollback

#### 5. Enhance `tests/smoke_test.sql` → add `tests/integration_test.sql`

- Verify row counts post-ETL (non-zero assertions)
- Validate procedure execution completes without error
- Check data quality log for failures
- Verify masking policies are applied correctly

#### 6. Add GitHub Environment configuration docs

Create `.github/ENVIRONMENTS.md` documenting:
- Required GitHub repository secrets per environment
- GitHub Environment protection rules to configure in Settings
- Required secrets: `SNOWFLAKE_ACCOUNT`, `SNOWFLAKE_USER`, `SNOWFLAKE_PASSWORD` (per-environment)
- For PROD: `SNOWFLAKE_PROD_ACCOUNT`, `SNOWFLAKE_PROD_USER`, `SNOWFLAKE_PROD_PASSWORD`

#### 7. Add workflow_dispatch for manual deployments and rollbacks

Allow operators to manually trigger a deployment to any environment (with approval gates still enforced by GitHub Environments).

1. **Local:** Run `bash scripts/deploy.sh --env=dev --dry-run` to validate script logic
2. **CI:** Push to a feature branch → confirm `ci.yml` lint passes
3. **QA:** Merge to `release/1.0` → confirm auto-deploy + smoke tests pass
4. **Pre-PROD:** Merge to `main` → confirm approval gate blocks until reviewer approves
5. **PROD:** Create tag `v1.0.0` → confirm 2-reviewer gate, deploy, smoke tests, notification
